# PRD: B2B/B2G 수출 매칭 커뮤니티 플랫폼

작성일: 2026-07-10
상태: 설계 진행 중 (미확정 영역은 19장에 명시)
관련 문서: DESIGN.md (디자인 요구사항)

---

## 0. 개발 규칙 (모든 산출물에 예외 없이 적용)

1. 하드코딩 금지
   - UI 텍스트: i18n 언어팩으로 관리
   - 비밀값: 환경변수로 관리
   - 상수/열거값: 중앙 모듈로 관리
2. 이모지 금지
   - 코드, 주석, 커밋, UI, 문서 전부
3. 설계 철학: 정책은 코드가 아니라 관리자 설정
   - 기능은 구현해 두고, 노출과 작동 여부는 관리자 스위치로 제어한다
   - 등급, 배지, 권한, 알림, 혜택, 메뉴 등 모든 정책 레이어에 일관 적용

---

## 1. 플랫폼 정체성

- 한국 제조사/공급사와 해외 바이어를 잇는 B2B/B2G 웹/앱 커뮤니티
- 주력 영역: 산업용(Industrial) 제품과 EPC+IPP(플랜트/발전)
- 소비재 영역 병행: Commercial 메뉴에서 화장품, 건강기능식품, 의류, 기타 잡화 취급. 플랫폼이 산업재 전용으로 보이지 않도록 톤 유지
- 플랫폼은 단순 게시판이 아니라 모든 접촉을 중개하는 신뢰 중개자
- 핵심 자산: 바이어 리스트와 제조사/공급사 네트워크
- 수익 모델 두 축: 유료 구독(인증 배지) + 운영사 부가 서비스(Services)

---

## 2. 회원 모델

### 2.1 가입

- 가입 방식은 관리자가 `일반 가입`, `추천 가입 전용`, `가입 일시 중지` 중 선택한다
- 이메일+비밀번호 가입만 제공하며 이메일 인증을 필수로 한다
- 이메일은 입력 중 형식·중복 여부를 확인하고, 가입 제출 시 서버에서 다시 검증한다
- 비밀번호는 10자 이상, 영문 대문자·소문자·숫자·특수문자를 각각 1개 이상 포함한다.
  입력 중 조건별 충족 상태를 표시하며 비밀번호 확인 입력은 두지 않는다
- 가입 시 역할 선택 없음. 제조사/공급사/바이어 구분 없이 모두 동등한 회원으로 시작
- 역할과 신뢰는 가입 후 배지 획득으로 선택적으로 쌓아가는 성장 경로

### 2.2 등급

- 초기 등급: 일반 회원 / 유료(인증) 회원
- 등급 체계는 고정이 아니라 관리자가 추가/편집 가능한 동적 구조

### 2.3 권한 매트릭스

- "등급(행) x 행동(열)" 구조의 권한표를 관리자가 한 화면에서 온/오프
- 예시 항목: 무료 회원의 문의 발신 허용 여부, 게시글 작성 가능 여부 등
- 배지는 권한에 관여하지 않는다 (3장 참조)

---

## 3. 배지 시스템

원칙: 배지는 권한이 아니라 신뢰 신호다.

### 3.1 제조사 배지 (역할 신호)

- 무료
- 회사정보, 사업자등록번호, 각종 인증서 등 서류 업로드 후 관리자 승인
- "이 회원이 무엇을 하는 회사인가"를 나타냄

### 3.2 인증 배지 (신원 신호)

- 유료 구독 + 간이 신원 확인 (신분증, 명함, 회사정보 등)
- 결제만으로 부여하지 않는다. 관리자가 신원 확인 후 부여
- 모든 역할(제조사, 공급사, 바이어)에 적용되는 전역 신뢰 신호
- 프로필과 게시글 등 모든 활동에 표시

### 3.3 공통 규칙

- 중복 보유 가능 (예: 제조사 배지 + 인증 배지)
- 관리자 재량으로 부여/회수 가능
- 배지 종류 자체도 관리자가 관리 (추가/편집)

---

## 4. 구독 관리

- 초기 결제: 계좌이체 입금 확인 후 관리자가 수동으로 구독 부여
- 부여 시 관리자가 구독 기간을 직접 설정 (예: 1개월, 1년)
- 기간 만료 시 인증 배지 자동 소멸 (시스템 자동 처리)
- 만료 전 갱신 안내 알림 자동 발송 (발송 시점은 알림 정책 스위치)
- 관리자는 기간 중이라도 수동 회수 가능
- PG 연동은 추후 과제. 구독 상태 데이터 구조는 동일하게 유지하고 결제 수단만 교체

---

## 5. 유료(인증) 회원 혜택

1. 글 수량: 무료 회원은 판매/요청글 수 제한 (수량은 관리자 설정), 유료 회원은 무제한
2. 미니 홈페이지
   - 구성: 회사소개, 회사 전경사진, 판매글 취합, 회사소개자료, 인증서자료
   - 자체 도메인 연결 가능
   - 연락처/이메일 노출 금지 원칙 동일 적용 (9장)
   - 모든 접촉은 플랫폼 문의 버튼으로 수렴
   - 외부 방문자의 문의는 가입 유도 후 진행
   - 홈페이지가 없거나 관리하지 못하는 회사에게 강력한 유료 전환 동기
3. 추천 노출: 추천 영역에 자동 노출, 노출 수량은 관리자 설정 (로테이션 규칙은 추후 확정)
4. 혜택 체계 자체가 관리자 추가/편집 가능한 동적 구조

## 6. 메뉴 및 게시판 시스템

### 6.1 동적 메뉴 시스템

- 메뉴는 하드코딩된 네비게이션이 아니라 관리자가 생성/관리하는 시스템
- 관리자 기능: 추가 / 편집 / 삭제 / 순서 변경 / 노출 온오프
- 메뉴 생성 시 타입을 선택하여 게시판 성격을 결정

### 6.2 게시판 타입

| 타입             | 용도        | 특징                                       |
| ---------------- | ----------- | ------------------------------------------ |
| 제품 게시판      | 판매글      | 견적 문의, 사양 필드 3층 구조, 대표 미디어 |
| 요청 게시판      | 요청글      | 마감 개념 (선택 마감일 + 수동 마감)        |
| 유연 필드 게시판 | 필드형 안내 | 관리자가 글마다 필드 추가/편집             |
| 안내 게시판      | 운영사 발신 | 공지, 서비스 안내                          |

### 6.3 게시판별 스위치

- 스위치 1. 작성 권한: 관리자 전용 / 회원 개방
- 스위치 2. 검수 모드: 검수 후 게시(기본값) / 즉시 게시
- 스위치 3. 메뉴 노출: 온 / 오프 (초기 빈 게시판 숨김 용도)
- 전역 원칙: 모든 회원 게시글은 어느 게시판이든 검수 후 게시가 기본

### 6.4 초기 메뉴 구성

| 메뉴       | URL         | 타입                                                | 작성 권한                                       |
| ---------- | ----------- | --------------------------------------------------- | ----------------------------------------------- |
| Commercial | /commercial | 제품 게시판 (화장품, 건강기능식품, 의류, 기타 잡화) | 회원                                            |
| Industrial | /industrial | 제품 게시판 (주력)                                  | 회원                                            |
| EPC+IPP    | /epc        | 제품 게시판 (주력)                                  | 초기 관리자 전용, 추후 회원 개방 (검수 후 게시) |
| RFQ & ITB  | /requests   | 요청 게시판                                         | 회원                                            |
| Events     | /events     | 유연 필드 게시판                                    | 초기 관리자 전용, 추후 회원 개방 (검수 후 게시) |
| Services   | /services   | 안내 게시판                                         | 관리자 전용                                     |
| Notices    | /notices    | 안내 게시판                                         | 관리자 전용                                     |

- Events: 박람회, 기업행사 안내. 행사마다 정보 구조가 다르므로 유연 필드 사용
- Services: 운영사가 직접 제공하는 부가 서비스 안내 (예: 태국 식약청 허가 대행, 할랄 인증, 틱톡샵 판매 지원, 정부 지원사업 컨설팅). 문의는 기존 중개 문의 체계를 사용하되 수신자가 관리자

### 6.5 게시판 페이지 구조 (모든 타입 공통)

- 목록 페이지 (리스트)
- 본문 보기 페이지 (상세)
- 작성 페이지
- 수정(편집) 페이지
- 삭제: 별도 페이지가 아닌 2중 확인 액션
- 내 글 관리 화면: 내가 쓴 글 + 검수 상태 + 임시저장 글 목록

### 6.6 제품 게시판 상세 규칙

- 가격 필드 없음. 모든 제품은 견적 문의로 통일. 상세의 핵심 행동 버튼은 문의하기 하나로 수렴
- 사양 필드 3층 구조
  1. 공통 기본 필드: 제품명, 카테고리, 사진 등
  2. 추천 필드 풀: 관리자가 정의한 필드 목록(원산지, 최소주문수량, 재질 등)에서 작성자가 선택 후 값 입력. 이름이 통일되므로 필터/검색에 활용 가능
  3. 자유 필드: 풀에 없는 특수 사양은 작성자가 항목명을 직접 만들어 추가
- 카테고리
  - 체계는 관리자가 관리 (추가/수정/순서)
  - 등록 시 카테고리 선택을 받아 데이터는 처음부터 축적
  - 바이어 화면의 카테고리 네비게이션은 초기 숨김, 관리자 노출 스위치로 제어
  - 제품이 쌓이면 스위치를 켜는 순간 소급 작업 없이 활성화

### 6.7 요청 게시판 상세 규칙

- 마감일: 선택 입력. 입력 시 날짜 도래에 따라 자동 마감
- 마감일이 없어도 작성자가 언제든 수동 마감 가능
- 빈 마감일은 상시 모집 중이라는 정직한 신호
- 마감된 글: 문의 버튼 비활성화 + 마감 상태 표시

### 6.8 대표 미디어

- 이미지 여러 장 업로드 가능, 그중 대표 이미지 1개 선택
- 대표를 동영상(링크)으로도 지정 가능
- 대표 미디어 사용처: 목록 썸네일, 상세 상단, OG 공유 미리보기
- 목록에서는 동영상도 썸네일 이미지로 표시 (자동 추출), 상세에서만 플레이어 재생

## 7. 에디터 (게시판 본문 작성)

### 7.1 기본 서식

- 제목/본문 스타일, 굵게, 기울임, 밑줄
- 목록(순서/비순서), 표, 인용, 구분선, 링크

### 7.2 미디어

- 이미지: 파일 업로드 방식. 본문 내 원하는 위치에 삽입, 삽입 시 미리보기 표시. 드래그 앤 드롭과 붙여넣기 지원
- 동영상: 업로드가 아닌 링크 임베드 방식 (유튜브 등). 본문 삽입 가능
  - 자동 재생: 브라우저 정책상 음소거 자동 재생이 상한선. 사용자 조작 후 소리 재생
  - 자동 재생 여부는 관리자 스위치로 제어
- 첨부파일: PDF 등 문서 첨부 (회사자료, 제품 카탈로그, 각종 인증서)

### 7.3 작성 보조

- 임시 저장: 수동 저장 + 자동 저장. 임시저장 글은 본인만 접근, 이어서 작성 가능
- 작성 중 이탈 방지 경고 (전역 2중 확인 원칙과 일치)
- 번역 보조: 한국어 초안을 영어로 제안, 작성자 확인 후 확정 (11장 언어 정책의 관리자 스위치)

### 7.4 업로드 제한

- 이미지/첨부파일 제한(개당 크기, 글당 개수)은 전 회원 공통, 값은 관리자 설정
- 등급별 차등은 글 수량 하나로 단순 유지

## 8. 문의 중개 시스템 (플랫폼의 핵심 문법)

### 8.1 완전 중개 원칙

- 회원 간 직접 연락 불가. 모든 문의와 답변은 관리자 검토 후 전달
- 관리자는 전달 시 피드백을 추가 작성할 수 있다
- 산업용/EPC 특성상 금액 견적과 협상도 플랫폼 메시지 안에서 진행
- 문의는 실시간 채팅이 아니다: 검수를 거치는 비동기 서신 교환이며, UI 은유는 채팅(말풍선, 즉시 전송)이 아니라 이메일/티켓 시스템(문의 건 단위, 진행 상태 타임라인, 답변 작성)이다

### 8.2 문의 흐름 (상태 절차)

1. 문의 발송됨
2. 관리자 검토 중
3. 글쓴이에게 전달됨
4. 글쓴이가 답변 작성
5. 관리자 답변 검토 중
6. 발신자에게 답변 전달됨

- 대화가 이어지는 동안 모든 왕복이 이 사이클을 반복
- 발신자와 수신자는 자기 문의/답변의 진행 단계를 추적 가능 (대기를 불안이 아닌 과정으로)

### 8.3 반려 처리

- 일반 반려 (내용 부족, 표현 문제): 사유 통보 + 수정 후 재발송 기회
- 정책 위반 반려 (연락처 교환 시도 등 플랫폼 우회): 위반 사실을 명확히 통보, 반복 시 제재 가능성
- 게시글 반려도 동일 패턴: 사유 통보 + 수정 후 재제출

## 9. 민감 정보 보호 (타협 불가 원칙)

- 연락처, 이메일 등 민감 정보는 관리자만 열람 가능
- 회원 간에는 어떤 경로로도 절대 비노출 (미니 홈페이지 포함)
- 유일한 공식 예외: 코디네이터는 자기가 직접 추천한 회원의 이메일까지 열람 가능 (16.4장)
- 이중 방어
  1. 애플리케이션 층: 화면과 API 응답에서 차단
  2. 데이터베이스 층: RLS로 본인과 관리자 외 접근 자체를 차단 (최종 방어선)
- 애플리케이션에 버그가 있어도 데이터가 나가지 않아야 한다
- 시스템이 자동 발송하는 이메일 알림은 회원 간 노출이 아니므로 이 원칙과 충돌하지 않음

### 12.1 비회원 접근 (그라데이션 공개)

- 상세: 상단부(제품명, 대표 이미지, 설명 도입부, 일부 사양) 공개, 그 아래는 그라데이션 페이드아웃 + 로그인 유도

### 12.2 SEO

- 사이트 전역 SEO: 관리자 직접 편집 (사이트 제목, 설명, 대표 이미지, robots.txt 정책)
- 개별 글/미니홈페이지 SEO: 작성 내용에서 자동 생성
  - 제목 → 메타 타이틀
  - 설명 도입부 → 메타 설명
  - 대표 미디어 → OG 이미지 (동영상 대표 시 썸네일 사용)
  - 이미지 alt: 제품명 기반 자동 생성
- sitemap: 승인된 공개 글 기준 자동 생성/갱신
- 공개된 상세 상단부가 검색 로봇의 색인 대상이 되어 판매글/요청글이 검색에 노출됨

## 16. 대시보드, UID, 추천 체계

### 16.1 UID

- 모든 회원은 가입 시 숫자 UID를 부여받는다 (예: 123456)
- UID는 개인 프로필의 공개 식별자로 사용

### 16.2 회원 대시보드

- 모든 회원은 개인 대시보드를 가진다
- 대시보드에서 1회용 추천 링크와 QR 코드를 필요할 때 생성한다
- 추천 링크 기본 유효기간은 7일이며 관리자는 1~30일 범위에서 변경한다
- 일반 회원은 활성 링크 3개, 인증 회원 5개, 코디네이터 10개를 기본 한도로 한다
- 링크는 생성자만 조회·폐기할 수 있고 사용·만료·폐기 상태를 확인한다
- 추천 성과(직접 추천 가입자 수) 표시는 기본 숨김, 관리자 스위치로 노출 전환

### 16.3 추천 체계

- 추천 링크는 1명만 사용할 수 있는 고난도 무작위 토큰으로 생성하고 원문 대신 해시를 저장한다
- 가입 신청 시 링크를 최대 24시간 `예약`하고 이메일 인증 완료 시 `사용 완료`한다.
  인증하지 않으면 계정과 예약을 정리해 링크를 다시 사용할 수 있게 한다
- 추천링크로 가입한 회원은 추천인의 하위로 연결되어 추천 트리를 형성한다
- 기존 회원은 새 추천 링크로 추천 관계를 변경할 수 없다
- 관리자는 회원 목록에서 전체 추천 트리를 열람

### 16.4 로그인 및 계정 보안

- 기본은 PC·모바일 복수 세션을 허용하고 관리자는 단일 세션 정책으로 전환할 수 있다
- 단일 세션에서는 가장 최근 로그인만 유지하며 이전 화면에 종료 이유를 안내한다
- 회원은 로그인 및 보안 화면에서 최근 기기·접속 시간·대략적 위치를 확인하고
  다른 모든 세션을 종료할 수 있다
- 새 기기, 새 국가, 짧은 시간의 반복 실패 등 위험 신호를 조합해 보안 이메일을 발송한다.
  IP 변경 하나만으로는 알림을 발송하지 않는다
- 로그인 보안 기록은 기본 90일 보관하며 본인과 관리자만 열람한다
- 관리자 계정은 MFA 적용을 운영 필수 조건으로 둔다

### 16.5 코디네이터 역할

- 관리자가 특정 회원에게 선택적으로 부여하는 역할
- 열람 범위: 자기가 직접 추천한 회원만 (하위 트리 전체가 아님, 손자 이하 비노출)
- 열람 정보: 직접 추천 회원의 목록과 이메일까지 노출
  - 민감 정보 보호 원칙(9장)의 공식 예외. RLS 정책도 "본인, 관리자, 직접 추천 관계의 코디네이터"까지로 확장
- 메시지 열람: 직접 추천 회원이 주고받은 문의/답변(관리자 피드백 포함)의 전문을 열람 가능
  - 오해의 소지를 없애기 위해 요약이 아닌 전문 열람
  - 스위치 구조: 전역(모든 코디네이터 일괄 허용/차단) 기본 + 개별 코디네이터 단위 예외 지정, 개별 설정이 전역보다 우선
- 코디네이터 배지: 기본 배지 세트에는 없음. 배지 종류는 관리자 관리 영역이므로 관리자가 추후 생성해 부여 가능

### 16.6 코디네이터 직접 메시지 채널

- 코디네이터는 자기가 직접 추천한 가입자와 개별 메시지를 주고받을 수 있다
- 기존 문의 중개(8장)와 달리 검수 대기 없이 즉시 직접 전달되는 유일한 메시지 채널
- 모든 메시지는 관리자가 자동으로 열람 가능
- 코디네이터와 가입자 화면에는 관리자 열람 여부를 표시하지 않는다
- 단, 약관에 "플랫폼 내 메시지는 운영 목적상 관리자가 열람할 수 있다"는 포괄 고지를 둔다 (표시는 없되 고지는 있는 상태, 17장 미확정 항목과 연결)

### 16.7 회원 네트워크 피드

- 회원은 UID와 공개 프로필 사진을 작성자 아이덴티티로 사용해 짧은 비즈니스 소식을 게시한다
- 게시글 본문은 최대 2,000자, 이미지는 최대 10장. 다중 이미지는 콜라주/갤러리로 표시한다
- 공개 읽기: 비회원도 `/feed`와 개별 공유 URL을 열람할 수 있다
- 인증 동작: 작성/수정/삭제, 좋아요, 댓글(최대 800자), 재게시, 공유 기록, 팔로우
- 작성자만 자기 게시글과 댓글을 변경/삭제할 수 있고, 관리자는 운영 목적의 삭제만 가능하다
- 피드 게시글은 랜딩의 네트워크 섹션과 회원 UID 프로필에도 노출한다
- 개별 게시글은 외부 SNS나 메신저에 공유할 수 있는 영구 URL을 가진다
- 피드는 거래 상품/구매 요청 게시판과 분리한다. 상품·요청은 검수·문의 흐름을 유지하고,
  피드는 회원 활동과 전문성을 알리는 소셜 레이어로 사용한다

## 17. 관리자 (통합 관제)

원칙: 관리자 대시보드는 이 프로젝트의 모든 기능을 관리할 수 있어야 한다. 회원이 보는 모든 화면, 모든 정책, 모든 스위치가 관리자 영역에서 제어된다. 코드 수정 없이 관리자 화면에서 운영이 완결되는 것이 목표다.

### 17.1 관리자 대시보드 (홈)

- 검수 대기 수 요약: 게시글 / 문의 메시지 / 배지 신청
- 신규 가입 현황, 구독 만료 예정 목록
- 각 관리 영역으로의 진입점

### 17.2 회원 관리

- 회원 목록
  - 기본 컬럼: 번호, 이름, UID, 이메일, 연락처, 역할, 배지 유무, 기타
  - 검색: 이름 / UID / 이메일
  - 필터: 등급, 배지, 역할, 가입일, 상태별
  - 목록 내보내기: CSV 등. 연락처 포함이므로 내보내기 행위는 감사 로그에 기록
  - 일괄 작업: 선택 회원 다중 처리 (알림 발송, 등급 변경 등)
- 회원 상세 (목록 클릭 시)
  - 기본 정보와 프로필 전체
  - 제조사 배지 보유 시: 미니홈피 주소 표시, 미니홈피 수정/편집 기능
  - 작성 리스트: 판매글 / 요청글 등 선택하여 표시
  - 회원 관리 액션: 회원 중지, 탈퇴 처리, 비밀번호 재발행
  - 관리자 메모: 관리자만 볼 수 있는 내부 메모
  - 추천인 정보: 누구의 추천으로 가입했는지, 추천 트리상 위치
  - 활동 이력: 최근 접속일, 로그인 기록, 문의 송수신 건수
  - 검수 이력: 게시글/메시지 반려 횟수와 사유 누적 (제재 판단 근거)
  - 구독 정보: 구독 상태, 기간, 만료일, 입금 확인 이력
  - 배지 신청 서류: 심사 시 제출 서류를 상세 화면에서 바로 열람
- 회원 상태 체계
  - 상태: 활성 / 중지 / 탈퇴. 중지 시 사유 기록 필수
  - 탈퇴 처리: 게시글/문의 데이터는 삭제가 아닌 익명화 보존 (거래 기록 분쟁 대비)

### 17.3 등급 및 권한 관리

- 등급 추가 / 편집 (동적 등급 체계)
- 권한 매트릭스: 등급(행) x 행동(열) 온오프를 한 화면에서 관리
- 무료 회원 글 수량 제한값 설정

### 17.4 배지 관리

- 배지 종류 추가 / 편집 (제조사, 인증 외 코디네이터 배지 등 신규 생성 가능)
- 배지 신청 승인 / 반려 (제출 서류 열람 포함)
- 재량 부여 / 회수

### 17.5 구독 관리

- 입금 확인 후 구독 부여, 부여 시 기간 설정
- 수동 회수
- 만료 자동 처리 모니터링 (배지 자동 소멸, 갱신 안내 발송 현황)
- 유료 혜택 항목 추가 / 편집 (동적 혜택 체계)
- 추천 노출 수량 설정

### 17.6 메뉴 및 게시판 관리

- 메뉴 추가(타입 선택: 제품/요청/유연 필드/안내) / 편집 / 삭제 / 순서 변경 / 노출 온오프
- 게시판별 스위치: 작성 권한(관리자 전용/회원 개방), 검수 모드(검수 후 게시/즉시 게시)
- 카테고리 체계 관리 (추가/수정/순서) + 카테고리 노출 스위치
- 추천 필드 풀 관리 (제품 사양 필드 정의)
- Services / Events 작성 (관리자 전용 게시판, 유연 필드 편집 포함)

### 17.7 검수 관리

- 게시글 검수 대기열: 미리보기, 승인 / 반려(사유 입력)
- 문의 메시지 검수 대기열: 전달 / 반려(사유), 전달 시 피드백 첨부 작성
- 코디네이터 직접 메시지 채널 열람

### 17.8 추천 체계 및 코디네이터 관리

- 회원 목록에서 전체 추천 트리 열람
- 코디네이터 역할 부여 / 해제
- 코디네이터 메시지 열람 스위치: 전역(일괄 허용/차단) + 개별 예외 지정 (개별이 전역보다 우선)
- 추천 성과(가입자 수) 회원 노출 스위치

### 17.9 알림 정책

- 알림 유형별 이메일 병행 발송 온오프
- 구독 만료 갱신 안내 발송 시점 설정

### 17.10 SEO 및 콘텐츠 노출

- 사이트 전역 SEO: 사이트 제목, 설명, 대표 이미지, robots.txt 정책
- 대표/OG 이미지 등 사이트 이미지 관리
- (개별 글 SEO는 자동 생성이므로 관리 대상 아님)

### 17.11 시스템 설정 (전역 스위치 모음)

- 번역 보조 기능 노출 온오프
- 동영상 자동 재생 온오프
- 인앱 브라우저 외부 전환 대상 행동 설정 + 온오프
- PWA 설치 배너 온오프, 재노출 주기
- 이미지/첨부파일 업로드 제한값 (개당 크기, 글당 개수)
- 쿠키 배너 문구 관리 (i18n 언어팩 연동)

### 17.12 감사 로그

- 어느 관리자가 언제 무엇을 변경했는지 기록
- 대상: 회원 상태 변경, 배지 부여/회수, 구독 처리, 스위치 변경, 데이터 내보내기 등
- 관리자가 2인 이상이 되는 시점부터 필수

## 18. PWA 및 디자인 기준

### 18.1 설치 유도 (PWA)

- 웹 자체를 앱처럼 설치하는 PWA 방식 (앱스토어 배포 없이)
- 처음 접속자와 미설치 사용자에게 설치 권유 표시
- 권유 강도: 방해하지 않는 하단 배너로만. 전면 팝업 사용 안 함
  - 닫으면 일정 기간 재노출하지 않음
- 플랫폼별 처리
  - 안드로이드/데스크톱 크롬: 네이티브 설치 프롬프트 활용
  - iOS 사파리: 자동 프롬프트가 없으므로 "공유 버튼 → 홈 화면에 추가" 안내 UI 자체 제작
- 배너 노출 온/오프와 재노출 주기는 관리자 설정

### 18.2 디자인 기준 (모바일 퍼스트)

- 기준 캔버스: 작은 모바일뷰. 앱과 같은 체감을 목표
- 반응형 범위: 작은 태블릿, 작은 모니터 PC까지 깨지지 않는 프레임 구조
- 스켈레톤 UI: 모든 로딩 구간에 적용 (14장 로딩 표시 원칙의 구체화)

### 18.3 모바일 앱 제스처

- 당겨서 새로고침 (pull to refresh)
- 갤러리형 레이아웃에서 좌우 스와이프
- 로딩 인디케이터 등 앱 표준 인터랙션

### 18.4 쿠키 허용 배너

- 첫 방문 시 쿠키 동의 배너 표시 (해외 바이어 대상 서비스이므로 GDPR 등 글로벌 규제 대응)
- 필수 쿠키 / 선택 쿠키(분석 등) 구분, 사용자가 선택 동의 가능
- 동의 상태 저장 후 재노출하지 않음, 설정에서 언제든 변경 가능
- 배너 문구는 i18n 언어팩으로 관리 (하드코딩 금지 원칙)
- 설치 권유 배너(18.1)와 동시 노출되지 않도록 우선순위 규칙 필요: 쿠키 배너 우선, 처리 후 설치 배너 노출
